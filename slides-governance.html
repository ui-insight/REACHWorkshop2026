Data Governance as the Foundation for Context Engineering

A Reveal.js outline for the first workshop teaching module.

Module goal

Participants should leave with one core idea

Context engineering is not only about retrieving more information. It is about deciding what information can be trusted, used, shared, and acted on inside an institutional workflow.

Opening frame

Better context is only useful if it is governed.

More documents, more tools, and more retrieval layers do not automatically produce better AI output. Ungoverned context can make answers sound more confident while becoming less safe.

Definition

What do we mean by context engineering?

  • Designing what the model can see
  • Designing what the model can use
  • Designing what the model should rely on
  • Designing when the model should abstain or escalate

Governance questions

Every new context source should trigger these questions

  • Where did this information come from?
  • Who is allowed to access it?
  • How current and reliable is it?
  • Is it sensitive, regulated, or institution-specific?
  • What human review is still required?

Research analytics lens

You already govern data — for dashboards and reports

You decide which data source is authoritative for a metric, how to version a KPI definition, who can see which dashboard, and what happens when a source system changes. AI governance is the same discipline — but the AI acts on the data, not just displays it.

Worked example

Proposal routing assistant

Imagine connecting a current policy PDF, outdated forms in a shared drive, exception-heavy email guidance, and informal staff notes.

The system now has more context, but not necessarily trustworthy context.

What to teach

Where governance enters the example

  • Which source is authoritative?
  • Which sources are historical or supplemental?
  • Who refreshes the source material?
  • What should the system cite back?
  • When must it escalate to a human?

Presenter note

Do not frame governance as bureaucracy

Frame it as the condition that makes automation safe, inspectable, and institutionally legitimate. Governance is what determines whether the system should answer, abstain, or route the question to a person.

Discussion

Questions for participants

  • Which local data sources are reliable enough to become AI context?
  • Which ones are too messy, sensitive, or incomplete?
  • Who defines what is authoritative in your workflow?
  • What governance work has to happen before automation expands?
  • What governance practices from your reporting work transfer directly to AI?

Bridge forward

Next modules build on this foundation

Once governance is clear, the workshop can move into layers of context engineering, structured data, retrieval, synthetic corpora, and response evaluation.
