AI4RA Workshop

Reproducibility, accuracy, and putting it to work

Led by Barrie Robison

Opening frame

Your data is already layered — context engineering makes it explicit

HERD data in one system, policies in another, departmental spreadsheets somewhere else, and dashboards pulling it together. Context engineering is how AI works with institutional data where it actually lives — the same integration challenge you already solve for reporting.

The stack

A practical context stack

  • System or role instructions
  • Task prompt
  • Examples and templates
  • Files and documents
  • Tools and actions
  • Retrieved and structured institutional data

What to teach

What each layer contributes

  • Instructions set tone, caution, and escalation boundaries
  • The prompt defines the current task and output
  • Templates keep responses consistent
  • Policy documents provide institutional authority
  • Retrieved tables add current operational detail
  • Human review protects edge cases and exceptions

Decision rule

Use the thinnest layer that gets the job done

If a template and a clearer prompt solve the problem, stop there. If a trusted PDF is enough, do not jump straight to retrieval. If the task depends on local judgment, the right layer may be human escalation instead of more automation.

Evaluation

Evaluating accuracy

  • AI sounds confident even when wrong — your job is to verify
  • Check facts against authoritative sources, not against what sounds right
  • Look for common failure modes: hallucination, outdated info, subtle bias
  • Design verification strategies proportional to the stakes

Lessons learned

The GRANTED project

  • What the AI4RA NSF project revealed about deploying AI across institutions
  • Shared data infrastructure matters more than model choice
  • Governance work has to precede automation work
  • Cross-institutional interoperability requires shared schemas, not just shared tools

Worked example

Proposal routing response

A faculty member asks about routing deadlines and required internal approvals for a proposal. A generic answer can sound polished while missing local forms, current deadlines, or special cases.

Better workflows combine role guidance, templates, policy, current data, and escalation rules.

Exercise

Have participants map their own stack

  • Pick one workflow people want to automate
  • Write the smallest useful prompt
  • List the other context needed for trust
  • Sort it into layers
  • Mark what is ready now and what still needs governance work

From data to action

Connecting organized data to analytics

  • Observable, governed data feeds analytics pipelines and reports
  • AI-assisted decision-making works when the data model is trustworthy
  • The lakehouse makes the data AI-ready; the AI makes the data actionable

Decision framework

Adoption and institutional fit

  • Automate: repetitive, low-stakes, output-driven tasks with clear validation
  • Augment: human-in-the-loop tasks where AI prepares and humans decide
  • Leave alone: process-critical work that requires full auditability
Callback: Remember Sequita the squirrel from Promptulus? Her domain is auditability — that's the question running underneath this entire framework. Can you audit the workflow? If yes, you can automate or augment it. If not, it stays in the "leave alone" zone until you can. The data lakehouse module showed you what trustworthy data infrastructure looks like; now Sequita's question is your litmus test for every workflow you take home.

Discussion

What do you need at your institution?

  • Which workflows only need better prompts or templates?
  • Which ones need files or retrieval before they are useful?
  • Which become risky once the system can take an action?
  • Where should a human remain part of the context stack?
  • When would AI add value beyond what your existing dashboard already delivers?

Wrap-up and Q&A

Workshop takeaway

What to take home

  • The most important model is the data model
  • Reproducibility comes from organization, not from AI
  • AI is a tool, not an authority — someone must always own the decision
  • Start with the thinnest layer that solves the problem reliably
